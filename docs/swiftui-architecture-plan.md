# SwiftUI Architecture Plan

This plan defines the implementation blueprint for TradeBill Canada before the Xcode project is created.

## Architecture Goals

- Keep v1 local-only.
- Keep UI fast and simple for field use.
- Separate calculations from views.
- Keep PDF rendering isolated.
- Keep StoreKit entitlement logic isolated.
- Use SwiftData first unless a concrete blocker appears.
- Use Decimal for money.
- Make the MVP easy to test without a backend.

## Suggested App Layers

```text
App
  App entry point, root navigation, dependency setup

Models
  SwiftData persisted models and non-persisted value types

ViewModels
  Screen state, form state, validation, user actions

Views
  SwiftUI screens and reusable components

Services
  Business logic that should not live in views

PDF
  PDF renderer, layout primitives, filename builder

Store
  StoreKit 2 product loading, purchases, entitlement checks

Persistence
  SwiftData container setup, seed data, migrations later

Utilities
  Formatting, rounding, IDs, date helpers
```

## State Management

Use SwiftUI-native state first:

- `@Query` for simple SwiftData lists.
- `@Environment(\.modelContext)` for persistence actions.
- `@State` for small local UI state.
- `@Observable` or `ObservableObject` view models for multi-step forms and async flows.

Avoid a global app state container unless a clear need appears.

## Data Ownership

Persisted models:

- BusinessProfile.
- Client.
- Document.
- LineItem.
- AppSettings.
- PurchaseState.

Non-persisted value types:

- TaxPreset.
- TaxComponent.
- MoneyCalculationResult.
- PDFRenderRequest.
- StoreProductDisplay.

Tax presets should be generated by code for MVP, then copied onto each document as a snapshot when selected.

## Navigation

Use a tab-free root for MVP:

```text
Onboarding / Business Setup
  -> Home
    -> Document Editor
    -> Client Manager
    -> PDF Preview
    -> Settings
    -> Paywall
```

The home dashboard should be the main hub. Do not add a complex sidebar/tab architecture for v1.

## Core Services

- `CalculationService`
- `TaxPresetService`
- `DocumentNumberService`
- `DocumentConversionService`
- `PDFRenderService`
- `PDFFilenameService`
- `PurchaseManager`
- `EntitlementGate`
- `ImageStorageService`

## Testing Strategy

Prioritize tests for:

- Money calculations.
- Tax component calculations.
- Document numbering.
- Estimate-to-invoice conversion.
- Free limit gating.
- PDF filename sanitization.

UI tests can wait until the app shell is stable.
